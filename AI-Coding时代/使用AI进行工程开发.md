# 一、方案设计阶段

1. 数据库表设计。最终产物：建表sql文件、表结构设计图（不要传统的严格意义上的ER图，要大概这种）<img src="C:\Users\DELL\AppData\Roaming\Typora\typora-user-images\image-20260702225531910.png" alt="image-20260702225531910" style="zoom: 25%;" />

   并且加上表之间的1:1或1:N的关系，表是单表还是分表。

2. 服务之间的调用关系，后端各个服务之间的调用关系，后端和前端之间的调用，后端调用mysql、redis等的关系，如果用到了mq，消息的发送、消费关系等。最终产物：（一张包含这些关系的）架构图。

3. 如果有某些业务逻辑比较复杂或流程较长，也可以再画流程图来说明。

4. 接口文档设计。接口文档包含三个部分：①各个接口的路径、请求参数、响应，参数和响应都要有json示例；②业务状态码③字段的枚举值（如果字段有枚举的话）。最终产物：一个md文件，三个一级标题分别是”接口格式”、“业务状态码”、“字段枚举值”。（该文档不描述接口的任何业务逻辑）

tips：

* 所有的画图都使用自然语言和cc或codex对话，安装draw.io的MCP server
* 数据库表和接口文档的设计，先把prd丢给ai，生成初版，再人工完善。

# 二、Coding阶段

1. 使用方案设计阶段的产物，用AI生成各个表对应的的model结构体、http router注册、各个请求的请求和响应对应的结构体、各个http router的handler，hander中完成解析请求参数、调用下层service函数/方法、组装和返回响应，各个service函数/方法留空
2. 人工在service函数中写该service函数的实现逻辑，通过注释和伪代码的方式。用注释写的实现逻辑至少包含：①参数校验②用使用哪个/哪些参数查询哪个/哪些表，要使用哪个/哪些参数怎么更新或插入表，在说明之后都要写上完整的sql，该sql要几乎接近可以直接运行③要调用哪个外部接口，参数怎么组装、响应内容怎么用。总之，越详细越好。

# 三、测试阶段

1. 把prd、表sql文件、接口文档丢给ai，生成调用各个接口的测试用例，测试用例是针对接口测试的，每个测试用例都要给出输入（请求参数）和预期输出
2. 让ai跑测试用，将测试结果输出到一个md文件中，包含每个测试用例的测试结果。
